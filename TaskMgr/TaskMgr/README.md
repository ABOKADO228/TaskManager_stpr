# TaskMgr

TaskMgr - учебный веб-прототип для управления расписанием, группой, домашними заданиями, заметками и комментариями. Проект построен на Python Bottle, HTML-шаблонах, обычном CSS и чистом JavaScript.

Сервер сейчас отвечает за маршруты, шаблоны и статические файлы. Пользователи, группы, роли и события в прототипе сохраняются в браузере через `localStorage`.

## Возможности

- Стартовая промо-страница `/`.
- Регистрация и вход на странице `/reg-auth`.
- Сохранение созданных аккаунтов в браузере.
- Создание учебной группы с ролью старосты.
- Присоединение к группе по коду в роли ученика.
- Переключение между группами, в которых состоит пользователь.
- Просмотр участников выбранной группы и их ролей.
- Ролевые права: староста редактирует и удаляет расписание и события, ученик просматривает и комментирует.
- Недельное расписание с занятиями, домашними заданиями, практическими и другими событиями.
- Личные заметки пользователя.
- Комментарии к событиям.
- Страница оформленных заказов `/orders` с загрузкой данных из JSON-файла.
- Серверная валидация формы заказа с сохранением введенных значений при ошибках.
- Страница активных пользователей `/active-users` для варианта 6.
- Серверная валидация формы активного пользователя с проверкой даты и телефона.

## Запуск

Установите Python-зависимости:

```powershell
pip install -r requirements.txt
```

Запустите сервер:

```powershell
python app.py
```

Откройте в браузере:

```text
http://localhost:5555
```

Хост и порт можно изменить через переменные окружения:

```powershell
$env:SERVER_HOST = "127.0.0.1"
$env:SERVER_PORT = "8080"
python app.py
```

## Основные файлы

`app.py` - точка входа Bottle-приложения. Настраивает путь к шаблонам, подключает маршруты, отдает файлы из `static/` и запускает сервер.

`routes.py` - маршруты:

- `/` - стартовая страница;
- `/reg-auth` - вход и регистрация;
- `/main` - основное рабочее пространство;
- `/orders` - оформленные заказы;
- `/active-users` - активные пользователи;
- `/static/<filepath:path>` - CSS, JavaScript, шрифты и другие статические файлы.

`views/index.tpl` - промо-страница сервиса.

`views/reg-auth.tpl` - форма регистрации и входа. Создает аккаунты в `localStorage.taskmgr_users` и сессию в `localStorage.taskmgr_session`.

`views/main.tpl` - интерфейс профиля, группы, расписания, заметок и комментариев.

`views/orders.tpl` - страница оформленных заказов: список объектов и форма добавления.

`views/active-users.tpl` - страница активных пользователей: список объектов и форма добавления.

`active_user_service.py` - Python-логика для загрузки существующих пользователей из файла, расчета рейтинга активности, фильтрации активных пользователей, сохранения и проверки формы.

`order_service.py` - Python-логика для загрузки заказов из файла, сохранения, сортировки и проверки формы.

`data/active_users.json` - файл с существующими пользователями и их метриками активности для страницы `/active-users`.

`data/orders.json` - файл с заказами, который используется страницей `/orders`.

`tests/test_active_user_validation.py` - unit-тесты проверки даты активности и телефона.

`tests/test_order_validation.py` - unit-тесты проверки даты и телефона.

`static/scripts/week-calendar.js` - основная клиентская логика: авторизация, группы, роли, события, заметки и комментарии.

`static/content/css/` - обычные CSS-файлы. SCSS/Sass в проекте больше не используется.

`package.json` - оставлен как метаданные npm-проекта. Скриптов сборки стилей больше нет.

## Работа со стилями

Проект использует обычный CSS без SCSS/Sass. Три файла ниже являются точками входа и содержат только `@import`:

- `static/content/css/auth-reg/preview.css` - стили стартовой страницы;
- `static/content/css/auth-reg/auth-reg.css` - стили входа и регистрации;
- `static/content/css/main/main.css` - стили главного рабочего экрана.

Основные стили разнесены по маленьким файлам:

- `static/content/css/auth-reg/preview/` - блоки стартовой страницы;
- `static/content/css/auth-reg/auth/` - блоки страницы входа;
- `static/content/css/auth-reg/base/` - базовые стили входа;
- `static/content/css/auth-reg/responsive/` - адаптивные правила входа и стартовой страницы;
- `static/content/css/main/base/` - базовые правила главного экрана;
- `static/content/css/main/layout/` - общий layout приложения;
- `static/content/css/main/components/` - кнопки, сайдбар, календарь, детали, формы и модальные окна;
- `static/content/css/main/responsive/` - адаптивные правила главного экрана.
- `static/content/css/active-users/` - стили страницы активных пользователей.
- `static/content/css/orders/` - стили страницы оформленных заказов.

Запуск `npm install` или `npm run scss` больше не нужен.

## Как пользоваться

1. Откройте `http://localhost:5555`.
2. Перейдите на страницу входа или регистрации.
3. Зарегистрируйте новый аккаунт или используйте тестовые данные:
   - `admin` / `1234`;
   - `user1` / `1111`.
4. После входа откроется `/main`.
5. Создайте группу, чтобы стать старостой.
6. Передайте код группы другому пользователю.
7. Другой пользователь может присоединиться по коду как ученик.
8. Если групп несколько, переключайтесь между ними в левой панели.
9. Участники выбранной группы отображаются в правой панели.
10. Староста добавляет, редактирует и удаляет события недели.
11. Ученик просматривает события и оставляет комментарии.

## Оформленные заказы

Страница доступна по адресу:

```text
http://localhost:5555/orders
```

На странице:

- перед просмотром нужно авторизоваться через `/reg-auth`;
- список заказов загружается Python-кодом из `data/orders.json`;
- отображаются только заказы текущего пользователя;
- новые заказы добавляются через POST-форму;
- новый заказ сохраняется с `owner_id` и `owner_name` текущего пользователя;
- обязательные поля: номер, автор/клиент, описание, дата, телефон;
- ошибки показываются на этой же странице;
- введенные значения не сбрасываются при ошибках;
- после успешного добавления выполняется redirect, форма очищается, заказ появляется в списке;
- сортировка идет по дате: свежие заказы выше.

## Активные пользователи

Страница варианта 6 доступна по адресу:

```text
http://localhost:5555/active-users
```

На странице:

- общий список пользователей загружается Python-кодом из `data/active_users.json`;
- активные пользователи выбираются автоматически по рейтингу активности;
- новые пользователи добавляются через POST-форму;
- обязательные поля: ник, описание, дата активности, телефон, созданные события, комментарии, заметки, группы;
- рейтинг считается по формуле: `события * 4 + комментарии * 2 + заметки + группы * 3 + бонус свежей даты`;
- в список активных попадают пользователи с рейтингом от `10` баллов;
- ошибки показываются на этой же странице;
- введенные значения не сбрасываются при ошибках;
- после успешного добавления выполняется redirect, форма очищается, пользователь появляется в списке;
- сортировка идет по рейтингу активности, затем по дате активности.

## Тесты

Запуск unit-тестов:

```powershell
python -m unittest discover -s tests
```

Тесты проверяют корректность ввода даты, телефона и расчет рейтинга активности.

## Данные в браузере

- `localStorage.taskmgr_users` - аккаунты;
- `localStorage.taskmgr_session` - текущая сессия;
- `localStorage.taskmgr_groups` - группы, участники, роли, события и комментарии;
- `localStorage.taskmgr_notes_<userId>` - личные заметки;
- `localStorage.isAuth` - флаг авторизации;
- `localStorage.username` - логин текущего пользователя;
- `localStorage.userRole` - вспомогательное поле роли;
- `localStorage.rememberMe` - признак сохранения входа;
- `sessionStorage.tempAuth` - временная сессия.
- `taskmgr_user_id`, `taskmgr_username`, `taskmgr_display_name` - cookie для серверной проверки доступа к `/orders`.

## Ограничения

- Данные хранятся только в браузере и только на текущем устройстве.
- Пароли в прототипе хранятся в открытом виде.
- Серверная база данных не подключена.
- Защита страницы заказов использует обычные cookie прототипа без серверной подписи.
- Роли проверяются на клиенте, а не на сервере.
- Удаление групп пока не реализовано.
