# Спецификация TaskMgr

## 1. Назначение

TaskMgr - учебный веб-прототип для управления учебными группами, расписанием, событиями недели, домашними заданиями, заметками, комментариями, оформленными заказами и активными пользователями.

Сервер на Bottle отдает HTML-шаблоны и статические файлы. Часть данных хранится на клиенте в `localStorage` и `sessionStorage`, а страницы `/orders` и `/active-users` используют JSON-файлы на стороне проекта.

## 2. Технологии

- Backend: Python, Bottle.
- Шаблоны: `views/*.tpl`.
- Стили: обычный CSS в `static/content/css/`.
- Клиентская логика основного рабочего пространства: `static/scripts/week-calendar.js`.
- Клиентская логика страниц `/orders` и `/active-users`: встроенные небольшие скрипты в шаблонах.
- Хранение заказов: `data/orders.json`.
- Хранение пользователей с метриками: `data/active_users.json`.
- Сборка стилей: не используется.

SCSS/Sass в проекте отсутствует.

## 5. Роли

- Гость видит стартовую страницу и может перейти ко входу.
- Авторизованный пользователь может создать группу или войти по коду.
- Авторизованный пользователь может переключаться между своими группами.
- Авторизованный пользователь может просматривать участников выбранной группы.
- Староста `leader` создает, редактирует и удаляет события группы.
- Ученик `member` просматривает события и оставляет комментарии.

## 6. Функциональность

- Регистрация сохраняет пользователя в `localStorage.taskmgr_users`.
- Вход создает `localStorage.taskmgr_session`.
- Вход также сохраняет cookie `taskmgr_user_id`, `taskmgr_username`, `taskmgr_display_name`.
- Создание группы назначает создателю роль `leader`.
- Присоединение по коду назначает роль `member`.
- События недели хранятся внутри объекта группы.
- Заметки сохраняются отдельно для каждого пользователя.
- Комментарии сохраняются внутри событий группы.
- Страница `/orders` доступна только авторизованному пользователю.
- Страница `/orders` выводит только личные заказы текущего пользователя.
- Страница `/active-users` выводит пользователей, прошедших порог активности.

## 6.1 Оформленные Заказы

Объект заказа содержит:

- `number` - номер заказа;
- `author` - автор или клиент;
- `text` - описание;
- `date` - дата в формате `YYYY-MM-DD`;
- `phone` - телефон;
- `owner_id` - id пользователя, создавшего заказ;
- `owner_name` - отображаемое имя пользователя.

Проверки формы:

- номер обязателен, уникален среди заказов текущего пользователя и содержит 3-20 допустимых символов;
- автор обязателен и не короче 2 символов;
- описание обязательно и не короче 10 символов;
- дата обязательна, соответствует формату `YYYY-MM-DD` и не позже текущей даты;
- телефон обязателен и содержит корректный российский номер.

Поведение страницы:

- GET `/orders` загружает заказы из `data/orders.json`;
- список фильтруется по `owner_id` из cookie;
- список сортируется по дате от новых к старым;
- POST `/orders` валидирует форму;
- при ошибках шаблон получает `errors`, `form`, `today`, `saved=False`;
- при успехе заказ сохраняется, затем выполняется redirect на `/orders?saved=1`;
- параметр `saved=1` включает уведомление об успешном добавлении;
- шаблон получает `today`, чтобы ограничить HTML-поле даты через `max`;
- список можно фильтровать на клиенте через поле поиска;
- кнопка `Очистить` сбрасывает форму.

## 6.2 Активные Пользователи

Объект пользователя содержит:

- `nick` - ник;
- `description` - описание активности;
- `active_date` - дата активности в формате `YYYY-MM-DD`;
- `phone` - телефон;
- `events_created` - количество созданных событий;
- `comments_count` - количество комментариев;
- `notes_count` - количество заметок;
- `groups_joined` - количество групп.

При выводе активных пользователей к объекту добавляется поле:

- `activity_score` - расчетный рейтинг активности.

Проверки формы:

- ник обязателен, уникален и содержит 3-24 допустимых символа;
- описание обязательно и не короче 10 символов;
- дата активности обязательна, соответствует формату `YYYY-MM-DD` и не позже текущей даты;
- телефон обязателен и содержит корректный российский номер;
- все метрики активности обязательны и должны быть неотрицательными целыми числами.

Формула рейтинга:

```text
события * 4 + комментарии * 2 + заметки + группы * 3 + бонус корректной даты
```

Активным считается пользователь с рейтингом от `10` баллов. Пользователи сортируются по рейтингу активности от большего к меньшему, затем по дате активности.

Поведение страницы:

- GET `/active-users` загружает пользователей из `data/active_users.json`;
- Python-код рассчитывает рейтинг и оставляет только активных пользователей;
- POST `/active-users` валидирует форму;
- при ошибках шаблон получает `errors`, `form`, `today`, `saved=False`;
- при успехе пользователь сохраняется, затем выполняется redirect на `/active-users?saved=1`;
- параметр `saved=1` включает уведомление об успешном добавлении;
- шаблон получает `today`, чтобы ограничить HTML-поле даты через `max`;
- список можно фильтровать на клиенте через поле поиска;
- форма показывает предварительный рейтинг до отправки;
- кнопка `Очистить` сбрасывает форму и пересчитывает предпросмотр рейтинга.

## 7. Хранение Данных

| Ключ или файл | Описание |
| --- | --- |
| `localStorage.taskmgr_users` | Аккаунты |
| `localStorage.taskmgr_session` | Текущая сессия |
| `localStorage.taskmgr_groups` | Группы, роли, события и комментарии |
| `localStorage.taskmgr_notes_<userId>` | Личные заметки |
| `localStorage.isAuth` | Флаг авторизации |
| `sessionStorage.tempAuth` | Временная сессия |
| `taskmgr_user_id` | Cookie с id пользователя для серверной фильтрации заказов |
| `taskmgr_username` | Cookie с логином пользователя |
| `taskmgr_display_name` | Cookie с отображаемым именем |
| `data/orders.json` | Файл заказов |
| `data/active_users.json` | Файл пользователей и метрик активности |

## 8. Документация В Коде

Python-файлы используют комментарии в стиле документации:

- краткое описание функции;
- `@param` для параметров;
- `@returns` для результата;
- `@throws` для исключений и особенностей обработки ошибок;
- `@note` для важных примечаний.

Этот стиль используется в `routes.py`, `order_service.py`, `active_user_service.py` и должен сохраняться при дальнейшей доработке проекта.

## 9. Ограничения

- Нет серверной базы данных.
- Пароли хранятся в браузере в открытом виде.
- Cookie авторизации в прототипе не подписаны серверным секретом.
- Проверка ролей выполняется на клиенте.
- Удаление группы не реализовано.

## 10. Критерии Приемки

- `/` загружает стартовую страницу без ошибок.
- `/reg-auth` позволяет войти тестовыми пользователями и создать нового пользователя.
- `/main` открывает рабочее пространство после входа.
- `/orders` перенаправляет неавторизованного пользователя на `/reg-auth?next=/orders`.
- `/orders` показывает только заказы текущего пользователя.
- `/orders` добавляет валидный заказ в `data/orders.json`.
- `/orders` показывает ошибки валидации без очистки формы.
- `/orders` показывает уведомление после успешного добавления.
- `/orders` позволяет искать по номеру, автору, описанию и телефону.
- `/orders` ограничивает дату сегодняшним днем.
- `/active-users` загружает `active-users.css` без 404.
- `/active-users` добавляет валидного пользователя в `data/active_users.json`.
- `/active-users` показывает ошибки валидации без очистки формы.
- `/active-users` показывает уведомление после успешного добавления.
- `/active-users` позволяет искать по нику, описанию, телефону и метрикам.
- `/active-users` показывает предварительный рейтинг активности.
- Unit-тесты выполняются через `python -m unittest discover -s tests`.
- В проекте нет папки `static/content/scss`.
- В `package.json` нет Sass-зависимостей и SCSS-скриптов.

## 11. Тестирование

Команда запуска:

```powershell
python -m unittest discover -s tests
```

Проверяются сценарии:

- корректная дата `YYYY-MM-DD`;
- некорректный формат даты;
- корректный российский телефон;
- некорректный короткий телефон;
- расчет рейтинга активности;
- фильтрация заказов по текущему пользователю;
- сохранение владельца нового заказа.
